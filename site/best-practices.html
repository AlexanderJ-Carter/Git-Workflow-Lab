📝 提交信息规范 重要

⚠️
为什么规范重要？ 清晰的提交信息让团队成员快速理解代码变更，也便于自动化工具生成 CHANGELOG。

推荐：Conventional Commits 格式

type(scope): description [optional body] [optional footer(s)]

常用 type 类型

类型 说明 示例
feat 新功能 feat: 添加用户登录功能
fix Bug 修复 fix: 修复登录超时问题
docs 文档更新 docs: 更新 API 文档
style 代码格式 style: 格式化代码
refactor 重构 refactor: 重构用户认证模块
test 测试 test: 添加单元测试
chore 杂项 chore: 更新依赖版本
  • 使用祈使句（动词开头，现在时）
  • 首字母小写，结尾不加句号
  • 保持简洁，正文必要时才写
  • 50 字符内描述提交

🌿 分支管理策略

常用分支命名规范

分支类型 命名格式 示例
功能分支 feature/ feature/user-login

热修复分支 hotfix/ hotfix/security-patch

✅ 日常操作规范

  • 🔄
    每天开始工作前先 pull
    确保本地代码是最新的，减少合并冲突
  • 📦
    切换分支前 stash 当前工作
    未完成的修改先暂存，避免丢失或携带到其他分支
  • 🎯
    一个提交只做一件事
    功能、修复、格式化分开提交，便于回溯和 Code Review
  • 📋
    使用 PR 进行代码审查
    不要直接 push 到 main，所有代码通过 Pull Request 审核合并
  • 🧹
    定期清理本地分支
    合并后删除已合并的本地分支，保持工作区整洁

🔒 安全规范 必须注意

🚫
永远不要做的事情：
  • ❌
    不要强制 push（git push --force）
    强制推送会覆盖他人代码，造成历史丢失。必要时先沟通。
  • ❌
    不要在公共分支执行 rebase
    rebase 会改写历史，影响团队其他成员的本地仓库。
  • ❌
    不要提交敏感信息
    密码、API Key、证书等敏感信息切勿提交到仓库，使用环境变量或 Secrets。
⚠️
敏感信息泄露了？ 立即：1) 修改所有暴露的密码/密钥 2) 从历史中清除 3) 通知安全团队

⚔️ 冲突解决指南

解决冲突的步骤

  1. 运行 git status 查看冲突文件
  2. 打开冲突文件，Git 会用 <<<<<< 标记冲突位置
  3. 保留需要的代码，删除冲突标记
  4. 运行 git add <file> 标记已解决
  5. 运行 git commit 完成合并
💡
最佳实践： 解决冲突时，优先与相关同事沟通，确保理解双方的修改意图。

↩️ 回滚操作指南

场景 推荐命令 说明
撤销未提交的修改 git checkout . 丢弃工作区所有修改
撤销已暂存的文件 git reset HEAD . 文件回到工作区
撤销本地最后几次提交 git reset --soft HEAD~n 保留修改在暂存区
撤销已推送的提交 git revert HEAD~n 创建新提交来撤销 安全
恢复误删的分支 git reflog 从日志中找回
⚠️
关于 git reset --hard： 此命令会丢失所有修改，不可逆。执行前请确保不需要这些修改，或先 git stash。

🤖 推荐工具配置

  • ⚡
    Git 补全
    Bash: 添加 git-completion；Zsh: 使用 oh-my-zsh git 插件
  • 🎨
    更友好的 diff
    使用 git difftool 配合 IDE 或 vimdiff
  • 📊
    git aliases（提升效率）
    建议配置：co = checkout, br = branch, st = status, unstage = reset HEAD --
  • 🔍
    .gitignore 模板
    使用 gitignore.io 生成适合项目的模板

📋 每日检查清单

  • 提交前检查 git status 和 git diff
  • 确认提交信息符合规范
  • 提交前确认没有敏感信息
  • 推送前确保本地代码已测试
  • 合并后及时删除已合并的分支
  • 定期执行 git gc 清理仓库
