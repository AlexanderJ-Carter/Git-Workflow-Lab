📄 常用配置模板

开箱即用的 Git 配置文件模板，复制即可使用

📄

.gitignore

忽略不需要纳入版本控制的文件，如编译产物、日志、临时文件等。

.gitignore 
# 操作系统文件
.DS_Store
Thumbs.db
*.swp
*~

# Node.js
node_modules/
npm-debug.log*
yarn-debug.log*
yarn-error.log*

# 构建产物
dist/
build/
*.egg-info/
__pycache__/
*.pyc

# IDE
.vscode/
.idea/
*.sublime-*

# 日志
*.log
logs/

# 环境变量
.env
.env.local
.env.*.local

使用方式

将文件保存为项目根目录的 .gitignore，支持全局忽略和仓库级别忽略。

⚙

.gitconfig

Git 全局配置文件，设置用户名、邮箱、别名、合并工具等。

~/.gitconfig 
[user]
    name = Your Name
    email = your.email@example.com

[alias]
    st = status
    co = checkout
    br = branch
    ci = commit
    lg = log --graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr) %C(bold blue)<%an>%Creset' --abbrev-commit
    undo = reset --soft HEAD~1
    amend = commit --amend --no-edit

[core]
    editor = code --wait
    autocrlf = input
    excludesfile = ~/.gitignore_global

[push]
    default = simple

[pull]
    rebase = false

[credential]
    helper = store

使用方式

运行 git config --global -e 编辑全局配置，或创建 ~/.gitconfig 文件。

提交信息规范

Conventional Commits 格式规范，使提交信息结构化，便于自动化工具处理。

Commit Message 
# 类型: 简短描述 (不超过50字)
# |类型|描述|
# |---|---|
# |feat|新功能|
# |fix|bug 修复|
# |docs|文档更新|
# |style|代码格式|
# |refactor|重构|
# |perf|性能优化|
# |test|测试相关|
# |chore|构建/工具|

# 示例:
feat: 添加用户登录功能
fix: 修复购物车数量计算错误
docs: 更新 API 文档

# 详细描述 (可选)
# 正文描述_what/why/how，72字符换行

# 相关 issue (可选)
Closes #123
Fixes #456

使用方式

可配合 commitizen 工具或 Git Hooks 强制执行规范格式。

📦 Git Hooks 模板

🛠

pre-commit Hook

提交前自动检查：代码格式、语法错误、敏感信息等。

.git/hooks/pre-commit 
#!/bin/sh

# 检查是否有敏感信息泄露
if git diff --cached --name-only --diff-filter=ACM | xargs grep -i "password\|secret\|api_key" 2>/dev/null; then
    echo "Error: 敏感信息不应被提交!"
    exit 1
fi

# 检查代码格式 (Node.js)
if [ -f "package.json" ]; then
    npx prettier --check "src/**/*.{js,jsx}" || exit 1
fi

# 运行单元测试
npm test || exit 1

exit 0

使用方式

保存为 .git/hooks/pre-commit 并添加执行权限 chmod +x .git/hooks/pre-commit

🎯 分支管理策略

推荐使用 Git Flow 或 Trunk-Based Development 等分支策略

Git Flow 工作流

适合有计划发布周期的项目，有明确的功能、发布、维护分支。

分支结构 
main (生产)     ─●──●──●──●──●──●──
                 \               /
develop (开发)   ─●──●──●──●──●──●
                  |             |
feature/        f1 f2         f3
                  |             |
release/                              r1 r2
                                  |

hotfix/                          h1
主分支只接受合并，不直接提交
不要在 main 分支上直接开发

Trunk-Based Development

适合持续交付的团队，所有开发在主干进行，使用短生命周期分支。

分支策略 
# 推荐策略
main ─●──●──●──●──●──●──●──●──
      │  │  │  │  │  │
      f1 │  │  │  │  └─ f4 (1-2天)
      │  f2 f3│  │  │
      └───────┘  │  └────── f5
                 └────────── f6

# 规则
- 分支生命周期 < 2 天
- 频繁集成到主干
- 功能开关控制发布
保持分支短小，频繁合并
避免长时间存在的分支

📝 提交规范

良好的提交习惯

  • 每次提交只做一件事
  • 提交信息清晰描述"做了什么"和"为什么"
  • 保持提交原子性（不可再分）
  • 先测试再提交
好的提交示例 
# 好的提交
feat: 添加用户头像上传功能
- 支持 JPG/PNG/GIF 格式
- 最大 2MB 限制
- 上传前进行图片压缩

fix: 修复登录页面验证码不刷新的问题
- 点击验证码图片时自动刷新

需要避免的做法

  • 提交信息过于笼统（如 "fix"）
  • 把多个不相关的更改放在一起提交
  • 提交未测试的代码
  • 在提交信息中解释如何修复
需要避免的提交 
# 需要避免的提交
update
asdf
fix bug
WIP
merged from origin

🔧 常用工作流

同步远程仓库

Bash 
# 获取远程更新并合并
git pull

# 或者分步执行
git fetch origin
git merge origin/main

# 使用 rebase 保持线性历史
git pull --rebase

整理提交历史

Bash 
# 合并最近3个提交
git rebase -i HEAD~3

# 压缩合并
git rebase -i HEAD~3
# 将 pick 改为 squash 或 s

# 修改最后一次提交
git commit --amend

撤销操作

Bash 
# 撤销工作区修改
git checkout -- file
git restore file

# 撤销暂存区
git reset HEAD file

# 撤销提交 (保留修改)
git reset --soft HEAD~1

# 撤销提交 (不保留修改)
git reset --hard HEAD~1

# 创建撤销提交
git revert HEAD
💡
提示

在团队协作中，优先使用 revert 而非 reset 来撤销公开提交的更改，这样可以保留完整的提交历史。

❓ 常见问题解答

精选高频问题，快速定位解决方案

根据需求选择不同的撤销方式：

  • git reset --soft HEAD~1 - 撤销提交，保留修改在暂存区
  • git reset --hard HEAD~1 - 撤销提交和修改，完全删除
  • git revert HEAD - 创建新的撤销提交（推荐用于已推送的提交）
⚠

如果是已推送到远程的提交，建议使用 git revert 而非 git reset，以避免影响他人。

合并冲突发生时，按以下步骤处理：

  1. 运行 git status 查看冲突文件
  2. 手动编辑冲突文件，保留需要的代码
  3. 冲突标记格式如下：
冲突文件 
<<<<<< HEAD
const a = 1;
=======
const a = 2;
>>>>>>> feature-branch
  1. 删除冲突标记，保留正确代码
  2. git add . 标记已解决
  3. git commit 完成合并

如果需要修改最后一次已推送的提交：

Bash 
# 修改最后一次提交
git commit --amend -m "新的提交信息"

# 强制推送到远程
git push --force-with-lease

# 注意：修改已推送的提交会改写历史，
# 在团队协作中请谨慎使用
⚠

修改已推送的提交会改写 Git 历史，可能影响其他协作者。建议只在个人分支上操作，或提前通知团队成员。

使用 git reflog 可以恢复误删的内容：

Bash 
# 查看操作历史
git reflog

# 从历史中恢复分支
git checkout -b branch-name HEAD@{n}

# 或者直接重置
git reset --hard HEAD@{n}

# 例如：恢复误删的分支
git reflog                         # 找到删除前的 HEAD
git branch recovered-branch HEAD@{1}
💡

Git 会在 reflog 中记录约90天的操作历史，即使是删除的分支或提交也有机会恢复。

merge 和 rebase 都是合并分支的方式，但有本质区别：

merge (合并)

  • 保留完整分支历史
  • 创建合并提交
  • 操作安全，不可破坏历史
  • 历史图可能复杂

rebase (变基)

  • 线性历史，更清晰
  • 不创建额外提交
  • 需要谨慎，不能在公共分支使用
  • 可能需要强制推送
使用场景 
# 本地分支整理，使用 rebase
git checkout feature
git rebase main

# 合并功能分支，使用 merge
git checkout main
git merge feature

📖 Git 术语表

常用 Git 术语中英文对照及解释

仓库

Repository / Repo

存储项目代码和版本历史的地方，包含 .git 目录。分为本地仓库和远程仓库。

git init, git clone

提交

Commit

对文件修改的快照，每次提交都会创建一个唯一的哈希标识。包含作者、时间、提交信息。

git commit -m "add feature"

分支

Branch

从主分支分离出来的独立开发线，允许在不影响主线的情况下进行开发。

git branch, git checkout -b

合并

Merge

将一个分支的修改整合到另一个分支。会产生一个新的合并提交。

git merge feature-branch

变基

Rebase

将一系列提交"移动"到另一个基点上，产生线性历史。用于保持提交历史整洁。

git rebase main

暂存区

Staging Area / Index

一个介于工作区和仓库之间的区域，用于准备要提交的修改。

git add file, git status

工作区

Working Directory / Working Tree

项目文件的实际目录，可以看到的文件内容。

编辑文件、查看修改

远程仓库

Remote

托管在网络上的仓库版本，通常用于团队协作和代码共享。

origin, upstream

拉取

Pull

从远程仓库获取并合并更新。相当于 fetch + merge。

git pull origin main

推送

Push

将本地仓库的提交上传到远程仓库。

git push origin main

检出

Checkout

切换分支或恢复文件。可以移动 HEAD 指针到指定提交。

git checkout branch-name

储藏

Stash

临时保存当前工作区的修改，以便切换到其他任务。

git stash, git stash pop

重置

Reset

移动 HEAD 和分支指针，可以撤销提交或修改暂存区。

git reset --hard HEAD~1

还原

Revert

创建一个新提交来撤销之前的修改，保留完整历史。

git revert HEAD

标签

Tag

为特定提交添加标记，通常用于版本发布。

git tag v1.0.0

冲突

Conflict

当两个分支修改了同一文件的同一位置时，合并无法自动进行，需要手动解决。

手动编辑、git add

引用日志

Reflog

记录所有 HEAD 和分支引用的变化历史，可用于恢复误操作。

git reflog

对象数据库

Object Database

Git 存储内容的地方，包含 blob、tree、commit、tag 四种对象类型。

.git/objects/
